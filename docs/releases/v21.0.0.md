# AI Launcher v21.0.0

v21 is the Trust & Flow / Command Deck release: a deeper security pass, a more deliberate visual system, and a stronger local release pipeline before public distribution.

## Highlights

- Command Deck visual refresh across the app shell, Command Center, Launcher, Workspaces, History, MCP, Updates, Admin, Onboarding, Analytics, Doctor, Prerequisites and Help.
- Safer credential handling through fail-closed secure storage backed by Windows Credential Manager.
- Backend-owned verified updater flow: the frontend no longer supplies raw download URLs for app updates.
- Runbooks 3.0 with explicit state machine, dry run, per-step approval, stop/retry/resume and secret-free resumable cursors.
- Unified workspace timeline for sessions, runbooks, MCP/security events and scoped activity.
- Versioned persistence, registry-backed backup/restore preview, bounded audit log and storage/capability guardrails.
- Critical workflow E2E coverage expanded to onboarding, workspace CRUD, providers, project profile writes, CLI session lifecycle, runbooks, MCP, backup, update failures and keyboard navigation.
- Release readiness contract and packaged Windows smoke harness added for local/CI release-candidate checks.

## Validation

Local v21 RC validation passed on Windows:

- `npx tsc --noEmit`
- `npm test`
- `npm run build`
- `npm run audit:storage`
- `npm run audit:capabilities`
- `npm run audit:prod`
- `npm run release:readiness`
- `npx playwright test`

Packaged build/smoke validation should be run before publishing final assets:

```powershell
npm run tauri build
npm run smoke:packaged -- --exe "src-tauri/target/release/ai-launcher.exe"
```

For a stable public release, run the smoke with `--require-signature` after signing is configured in CI.

## Notes

- No telemetry is introduced; local credentials, profiles, history and scans remain local unless the user explicitly exports or launches an external tool.
- Unsigned local builds may still trigger Windows SmartScreen. Public stable releases should use the signing workflow documented in `docs/SIGNING.md`.
